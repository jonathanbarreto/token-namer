Framework at a glance

Primitive <property>/<identifier> <property>/<group>/<identifier>
Semantic <role>/<element>/<variant?>/<emphasis?>/<state?>/<context?>
Component <component>/<part>/<property>/<variant?>/<state?>/<context?>
  1. Pick a tier to match your intent.
  2. Fill fields in order; optional fields appear as you go.
  3. Use context for inverse/on-media and state for interaction.

Best practices

  • Keep primitives literal; never encode meaning or usage.
  • Prefer semantic tokens for intent and theming decisions.
  • Only add component tokens where you need stable override points.

Primitive tokens

Literal values aligned to DTCG properties. No meaning or usage baked in.

<property>/<identifier>
<property>/<group>/<identifier>

Examples

color/blue/500
color/blue/500-a20
dimension/space/16
fontFamily/brand/brand-sans

Semantic tokens

Intent-first naming that maps to primitives and adapts per context.

<role>/<element>/<variant?>/<emphasis?>/<state?>/<context?>

Examples

surface/background/raised
content/text/primary/inverse
action/background/primary/hover

Role replaces the old domain term. Context captures inverse/on-media needs.

Component tokens

Component-specific overrides without a prefix.

<component>/<part>/<property>/<variant?>/<state?>/<context?>

Examples

button/container/background/primary/enabled
text-field/field/border-color/invalid

Use components for stable override points and anatomy-specific needs.

Input rules

  • Inputs normalize to kebab-case. Primitive properties keep camelCase when needed (e.g., fontFamily).
  • Optional fields appear based on prior selections and follow framework order.
  • Use context for inverse/on-media needs; use state for interaction (enabled, hover, pressed).
